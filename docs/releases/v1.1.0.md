## Main Deliveries

### Webhook security and request filtering

- Added `ALLOWED_CHAT_IDS` validation at the start of the FastAPI webhook flow.
- Stopped unauthorized chats before any Telegram download or Microsoft Graph work is triggered.
- Added chat metadata to the Telegram schemas so the webhook can identify the originating conversation.

### Telegram feedback loop

- Added a Telegram client helper for outbound `sendMessage` calls.
- Added success feedback after a receipt is forwarded.
- Added explicit user feedback for missing documents and invalid PDF submissions.

### Documentation and operations

- Updated the README to explain the new whitelist and destination email configuration.
- Updated the architecture notes to include the security validation and chat-feedback flow.
- Updated the operations guide with Cloud Run FinOps settings such as `--max-instances` and `--concurrency`.

## Compatibility

- Python runtime target: 3.11+
- Existing deployments must now define `ALLOWED_CHAT_IDS` for the webhook to start processing authorized chats.
- Existing deployments must continue to define `TELEGRAM_BOT_TOKEN`, `MICROSOFT_CLIENT_ID`, `MICROSOFT_REFRESH_TOKEN`, and `DESTINATION_EMAIL`.

## Recommended Deployment Procedure

1. Build the new image from the repository root.
2. Publish the image to the configured container registry.
3. Deploy the image to Cloud Run with `ALLOWED_CHAT_IDS` present in the environment.
4. Confirm the configured `DESTINATION_EMAIL` still points to the intended mailbox.
5. Verify the Telegram webhook continues to point to the final `/webhook` endpoint.

Reference operational guide: `docs/operations.md`.

## Post-Deploy Smoke Test

1. Confirm the Cloud Run revision starts successfully.
2. Send a PDF from an authorized Telegram chat and verify that the email is delivered.
3. Confirm the originating chat receives the success message.
4. Send a non-PDF document from an authorized chat and confirm the error message is returned.
5. Confirm an unauthorized chat is silently ignored and does not trigger the email workflow.

## Rollback

- Roll back to `v1.0.0` if the whitelist or Telegram feedback changes break the production flow.
- After rollback, re-check the runtime environment configuration before attempting a new deployment.
